496. Next Greater Element I


class Solution {
public:
    vector<int> nextGreaterElement(vector<int>& nums1, vector<int>& nums2) {
        vector<int> result;
        
        // mapping 就是我們的「筆記本」
        // Key: 矮個子 (被踢掉的人)
        // Value: 他的下一個高個子 (踢他的人)
        unordered_map<int, int> mapping;
        
        // st 就是我們的「等候室」，裡面永遠保持著「由大到小」的單調性
        stack<int> st;
        
        // ==========================================
        // 階段 1：高個子終結者大屠殺 (建立 Hash Map)
        // ==========================================
        for(int i = 0; i < nums2.size(); i++){
            
            // ⚔️ 連續踢人機制：
            // 只要房間不是空的，而且「現在這個人 (nums2[i])」比「站在門口的人 (st.top())」高
            while(!st.empty() && st.top() < nums2[i]){
                
                int temp = st.top(); // 抓出站在門口的最矮的人
                
                // 記錄在筆記本：這個矮個子 (temp) 遇到的高個子就是 nums2[i]
                mapping[temp] = nums2[i]; 
                
                // 他得到答案了，把他請出等候室
                st.pop(); 
            }
            
            // 踢完前面比自己矮的人之後，自己乖乖走進等候室排隊
            st.push(nums2[i]); 
        }
        
        // ==========================================
        // 階段 2：拿著名單查筆記本
        // ==========================================
        for(int i = 0; i < nums1.size(); i++){
            
            // 檢查名單上的人 (nums1[i]) 有沒有被記錄在筆記本裡？
            if(mapping.count(nums1[i]) != 0 ){
                // 有的話，把查到的高個子加進答案裡
                result.push_back(mapping[nums1[i]]);
            }
            else{
                // 沒有的話 (代表他到最後都沒遇到比他高的人)，填入 -1
                result.push_back(-1);
            }
        }
        
        return result;
    }
};

###  大哉問一：為什麼這題非得用 Stack (堆疊) 不可？

資料結構那麼多（陣列、Queue、Set...），為什麼偏偏是 Stack？
這要回到 Stack 最核心的物理特性：**後進先出 (LIFO - Last In, First Out)**。

想像一下這個情境：排隊時，你剛剛才走進等候室，你是「最新鮮」的（Last In）。
當下一個高個子走進來時，他第一眼看到的是誰？**絕對是剛走進來的你！**
他不可能跳過你，先去檢查三天前就坐在等候室最深處的那個人。

* **Queue (先進先出)：** 就像買手搖飲，排最前面的人先結帳。這不符合我們要「回頭看最近一個矮個子」的需求。
* **Stack (後進先出)：** 就像洗盤子，最後疊上去的盤子最先被洗。這完美契合了**「最新進來、還沒找到答案的矮個子，要優先被新來的高個子檢驗」**的邏輯！




### 🕵 大哉問二：以後怎麼一眼看穿「這題要用 Stack」？

面試時看到新題目，要怎麼立刻觸發「拔出 Stack 這把武器」的直覺？
請把你大腦裡的雷達設定好，只要題目敘述出現以下 **3 種關鍵字或情境**，高機率就是 Stack 題：

#### 特徵 1：尋找「最近的」某個特定關係 (Next / Previous)
* **關鍵字：** Next Greater, Next Smaller, Nearest, 第一個比我大/小的...
* **為什麼：** 因為 Stack 的 `top()` 永遠握著「距離現在最近」的歷史紀錄。
* **經典題：** 今天這題 496. Next Greater Element、739. Daily Temperatures (算要等幾天溫度才會變高)。

#### 特徵 2：具有「配對 / 消除」機制的題目
* **關鍵字：** Valid Parentheses (括號配對), 消除連續重複字元, 絕對路徑化簡 (`/a/../b`)。
* **為什麼：** 遇到右括號 `)`，你要找的是「最近剛出現」的左括號 `(` 來互相抵銷。後進先出的 Stack 是唯一解。
* **經典題：** 20. Valid Parentheses (這題如果還沒寫過一定要寫，超級經典！)。

#### 特徵 3：歷史紀錄需要保持「單調性」 (進階：單調棧)
* **情境：** 如果你發現某些歷史資料「一旦遇到某個條件，就再也沒有利用價值，可以直接丟掉」。
* **例如這題：** 一旦遇到高個子，前面的矮個子就找到答案了，他們就「失去價值」可以直接從 Stack 裡彈出去 (pop)，這讓 Stack 裡永遠保持著大到小的單調性。
* **經典題：** 84. Largest Rectangle in Histogram、42. Trapping Rain Water (這些是 Hard 題，但核心思想就是你今天學會的單調棧！)。
